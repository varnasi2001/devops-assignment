## Minikube instead of kind / cloud

I Wanted a local cluster so this repo runs on any machine with Docker without cloud credentials. Between kind, k3d, and Minikube:

- Minikube's Terraform provider (`scott-the-programmer/minikube`) tracks the cluster as a proper Terraform resource, so `terraform destroy` cleans up.
- Minikube's built-in addons (`metrics-server`, `default-storageclass`) are one string in `main.tf` each.
- Ingress needs `sudo minikube tunnel` on macOS/Linux which is a small annoyance but standard for local setups.

## Helm layout: templates + overrides

Every chart definition lives under `helm-templates/`. 
Every per-cluster tweak lives under `helm-overrides/<app-name>/`. 

The template folder is common code; the overrides folder is where you change values per environment.

Same pattern for our own charts (articles-backend, mongodb) — templates under `helm-templates/<name>/templates/`, overrides in `helm-overrides/<app-name>/custom-values.yaml`.

Terraform's `helm_release` reads both:
```hcl
chart  = ".../helm-templates/argo-cd-9.5.21"
values = [file(".../helm-overrides/argo-cd/custom-values.yaml")]
```

## MongoDB: custom chart, not Bitnami

Bitnami's chart works fine in prod but has 40+ template files. I wanted the replica-set init pattern to be readable — six templates in `helm-templates/mongodb/`. The keyfile + root password are generated at install time via Helm's `randAlphaNum`, and a post-install Job runs `rs.initiate()` once all pods are ready. The Job is idempotent — it checks `rs.status()` before initiating.

Trade-off: no operational maturity. For real prod use Bitnami or the MongoDB Operator.

## Anti-affinity: soft, not hard

3-replica workloads on a 3-worker cluster — hard anti-affinity means one pod evicted from a node stays Pending because there's no room for it elsewhere. Soft still spreads them but degrades gracefully. In a cluster with N+1 workers, hard is fine.

## PDB: `maxUnavailable: 1`

Same effect as `minAvailable: 2` for a 3-replica set, but reads more directly — "never disrupt more than one at a time". For mongo this also means quorum stays intact (2/3) during any drain.

## Probes: startup + readiness + liveness

Startup gives the container up to 150s (5s × 30) to come up before liveness kicks in. Otherwise a slow cold start (npm require + mongo connect) would get killed. Readiness checks the mongo connection state — kubectl only routes traffic to pods that are actually connected.

## Mongo connection: all three seeds

```
mongodb://user:pass@mongodb-0.mongodb-headless:27017,mongodb-1...,mongodb-2.../articles?replicaSet=rs0
```
## Secrets

Mongo root password + keyfile are generated by Helm at install time (`randAlphaNum`). Never committed. The backend reads them via `secretKeyRef`. The `deploy.sh` script copies the secret from `mongodb` namespace to `articles` namespace with `kubectl | sed | kubectl apply` — real deployments should use ExternalSecrets or SealedSecrets instead of this shell pipe.

## Observability: kube-prometheus-stack

VictoriaMetrics stack is what I work with at Meesho and it's great, but for a single-app demo it's four StatefulSets/Deployments (agent + insert + select + storage). One `kube-prometheus-stack` chart gives me Prometheus + Grafana + node-exporter + kube-state-metrics with one helm release, plus default K8s dashboards baked in. Simpler wins here.

Backend discovery uses `ServiceMonitor` (CRD from the stack) rather than pod annotations. Cleaner selectors and lets Prometheus follow the standard label conventions.

## GitOps: app-of-apps + two-source pattern

`argocd/master-argo-app.yaml` is the root App. Its source is `argocd/applications/` — a directory of child Apps. Each child App uses the two-source pattern:
- source 1: the git repo at `ref: values` — points at the whole repo for value-file lookups
- source 2: the git repo at `path: helm-templates/<chart>` — the chart itself, with `valueFiles: [$values/helm-overrides/.../custom-values.yaml]`

Adding a new workload = drop a new file under `argocd/applications/`. Argo CD picks it up on next sync.
